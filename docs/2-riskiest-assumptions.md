# Step 2 — Riskiest Assumptions Assessment (RAT)

Метод:

- Классификация рисков: value / usability / feasibility / viability (по SVPG).
  :contentReference[oaicite:0]{index=0}
- Скоринг: **risk score = impact × (10 − confidence)** (0..100).
  :contentReference[oaicite:1]{index=1}
  - impact: насколько это “убьёт” V1, если ложь (1..10)
  - confidence: сколько у тебя уже есть доказательств (0..10)

## 2) Детальная оценка + минимальные тесты (1–3 дня на каждый)

### Assumption 1 — “Кнопки + видео = диалог/интимность”

Риск:

- **Value**: людям может быть неинтересно/неестественно.
- **Usability**: ощущение “меня ведут по рельсам”, “кринж”, потеря агентности.
  :contentReference[oaicite:8]{index=8}

RAT-тест (самый быстрый):

- **Wizard-of-Oz прототип**: видео партнёра (записанное или полуживое), 6–10
  коротких сцен, на каждом шаге 2–4 кнопки-реплики.
- Метрики (pass/fail):
  - **Completion rate**: ≥ 60% доходят до конца мини-сцены (3–5 минут).
  - **“Agency score”** (1..7): медиана ≥ 5 (“я выбирал, а не меня везли”).
  - **“Cringe score”** (1..7): медиана ≤ 3.
  - Повтор: ≥ 30% хотят сразу вторую сцену.
- Решение по итогу: если fail → меняешь механику (больше свободы выбора / меньше
  “готовых фраз” / другой темп) до того, как трогать остальное.

---

### Assumption 2 — “Ожидание в очереди не убивает конверсию”

Риск:

- **Viability**: дисбаланс сторон → отсутствие ликвидности (liquidity) → пустой
  продукт.
- **Usability**: ожидание ощущается дольше реального, особенно если оно
  **неопределённое** или “несправедливое”. :contentReference[oaicite:9]{index=9}

RAT-тест:

- **Симуляция ожидания** в прототипе: искусственно задаёшь wait time (0 / 30с /
  2м / 5м) и смотришь отвал.
- Добавляешь “управление восприятием ожидания”:
  - показать **прогноз** и прогресс (known finite waits),
  - “занять” ожидание микро-действием (occupied time feels shorter),
  - убрать тревогу (объяснение “почему ждёшь”).
    :contentReference[oaicite:10]{index=10}
- Метрики (pass/fail):
  - Drop-off в очереди при 2 мин: ≤ 35%
  - Drop-off при 5 мин: ≤ 55% (если выше — очередь токсична)
  - Если “прогноз + прогресс” снижает drop-off минимум на 15 п.п. — это
    must-have механика.

---

### Assumption 3 — “Сцены синхронизируются гладко в браузере”

Риск:

- **Feasibility**: сетевые задержки, джиттер (jitter), потери пакетов, дрейф
  таймингов.
- **Usability**: отсутствие подтверждения действия (“я нажал — ничего”)
  разрушает чувство контроля. :contentReference[oaicite:11]{index=11}

RAT-тест (технический POC):

- Два браузера + канал синхронизации (WebRTC data channel / WebSocket).
- Протокол: **каждое действие = событие с id + ack**, локальный optimistic UI +
  откат при таймауте.
- Наблюдаемость: собираешь RTT (Round Trip Time), jitterBufferDelay, packet loss
  (через WebRTC stats / логи). :contentReference[oaicite:12]{index=12}
- Метрики (pass/fail):
  - P95 “click → ack” ≤ 250 ms на нормальной сети
  - P95 рассинхрон состояний ≤ 300 ms
  - 0 “тихих фейлов”: любое действие либо подтверждено, либо видно как “не
    доставлено”
- Важный факт: нижняя граница задержки упирается в RTT/2; ты не “оптимизируешь
  до нуля”, ты проектируешь UX поверх физики.
  :contentReference[oaicite:13]{index=13}

---

### Assumption 4 — “Текст баблом усиливает, а не перегружает”

Риск:

- **Usability**: конкуренция за внимание между видео, кнопками и текстом
  (split-attention).
- **Когнитивная нагрузка** (cognitive load): дублирование каналов может ухудшать
  восприятие (redundancy). :contentReference[oaicite:14]{index=14}

RAT-тест (вместе с Assumption 1):

- A/B: (A) баблы всегда, (B) баблы только ключевые слова / после выбора / по
  тапу.
- Метрики (pass/fail):
  - Ошибки выбора / “не понял что происходит”: B должно быть не хуже A
  - Субъективная перегруженность (1..7): медиана ≤ 3
  - Время до первого выбора (time-to-first-choice) не должно расти > 20%

---

### Assumption 5 — “Сценаристы редактируют без кода и не ломают граф”

Риск:

- **Feasibility**: редактор + валидация + версионирование.
- **Viability**: без этого контент масштабируется болью и ручным QA.
  :contentReference[oaicite:15]{index=15}

RAT-тест:

- **Мини-контент-модель** (structured content): Scene, Choice, Line, Condition,
  NextScene.
- Простейший редактор форм (schema-driven): поля + ограничения + предпросмотр +
  “проверка связности графа”.
- Дать 1–2 сценаристам 3 задачи:
  1. добавить сцену,
  2. поменять варианты выбора,
  3. сделать развилку и вернуть в общий узел.
- Метрики (pass/fail):
  - ≥ 80% задач сделаны без твоего вмешательства
  - 0 “сломанных билдов”: редактор блокирует невалидные изменения
  - Среднее время задачи ≤ 15 минут

---

## 3) Рекомендованный порядок на 2 недели (без расписывания по дням)

1. Прогнать пользовательский RAT на **(1)+(4)**.
2. Параллельно поставить тест ожидания **(2)** (хотя бы симуляцией).
3. Ранний POC синхронизации **(3)** (иначе UX окажется фантазией).
4. Если V1 зависит от контента — поднять минимальный “неубиваемый” редактор
   **(5)**.
